# 12 长期记忆产品化实现记录

## 当前阶段

本阶段先做长期记忆产品化的核心闭环：

```text
1. 记忆注入组装
2. 运行结束后的规则候选捕获
3. 记忆审核接口
4. 前端记忆管理面板
```

暂时不做：

```text
LLM 记忆抽取
复杂语义冲突判断
Session 摘要压缩
Skill Memory
```

## 为什么先做这两步

当前项目里已经有：

```text
memory_entries
memory_conflicts
MemoryStatus
MemoryEntryRepository
MemoryConflictRepository
```

但是之前记忆逻辑还停留在：

```text
AgentRunContextBuilder 直接查 ACTIVE 记忆
然后把记忆拼到 system prompt
```

问题是：

```text
记忆筛选策略散落在 runtime 里
没有统一注入预算
运行结束后不会自动产生候选记忆
长期记忆生命周期还没有入口
```

所以第一轮先把两个入口打通：

```text
运行前：ACTIVE 记忆怎么注入
运行后：用户消息怎么变成候选记忆
审核时：候选和冲突怎么进入 ACTIVE / REJECTED / DISABLED
```

## 记忆注入组装

新增长期记忆上下文组装器。

职责：

```text
加载当前 workspace + user 下的 ACTIVE 记忆
过滤空内容
限制注入数量
限制注入总字符数
限制单条记忆长度
输出稳定的 prompt 片段
```

当前预算：

```text
最多 12 条 ACTIVE 记忆
总字符数最多 1800
单条记忆最多 240 字符
```

注入格式：

```text
- [PROJECT_CONSTRAINT] 不要修改 generated 目录
- [WORKING_STYLE] 以后先讲设计再改代码
- [PROJECT_FACT] 当前项目使用 Maven 启动
```

如果没有可注入记忆：

```text
- 暂无可注入的长期记忆。
```

关键原则：

```text
只有 ACTIVE 记忆允许注入。
PENDING / CONFLICT / REJECTED / DISABLED 都不能影响 Agent。
```

### 隐式注入

记忆注入后应该表现为 Agent 的行为约束，而不是让 Agent 在回答中主动强调“长期记忆中记录了什么”。

因此 prompt 中不再使用面向用户的“长期记忆”标题，而是改为：

```text
已知偏好与项目约束
```

并明确要求：

```text
按偏好和约束行动即可。
不要主动提到“记忆”“长期记忆”“已记录”“记住了”等系统实现细节。
只有用户明确询问你记得什么，或要求确认保存结果时，才解释记忆内容。
```

这样做的原因：

```text
记忆是上下文治理能力，不应该污染正常回答。
用户要的是 Agent 行为变好，不是每次都听 Agent 复述内部状态。
```

## 记忆捕获

新增长期记忆捕获服务。

它在一次 Agent Run 正常完成后执行：

```text
用户消息
助手最终回答
workspaceId
agentId
conversationId
userMessageId
```

第一版只从用户消息里做规则捕获，不调用大模型。

原因：

```text
规则捕获稳定、可解释、便于面试讲清楚。
LLM 抽取后续可以作为增强，但不能作为第一版唯一来源。
```

## 当前捕获规则

### 项目约束

触发词：

```text
不要改
不要修改
别改
禁止修改
不能修改
不要动
```

类型：

```text
PROJECT_CONSTRAINT
```

例子：

```text
记住：以后不要修改 generated 目录。
```

如果用户明确说“记住”，且没有冲突：

```text
status = ACTIVE
```

否则：

```text
status = PENDING
```

### 协作风格

触发词：

```text
我喜欢
我希望
回答风格
你以后
以后你
直接给
先讲
先说
中文注释
```

类型：

```text
WORKING_STYLE
USER_PREFERENCE
```

例子：

```text
记住：以后改代码必须写中文注释。
```

### 项目事实

触发词：

```text

当前项目
启动命令
测试命令
技术栈
模块
使用的是
用的是
```

类型：

```text
PROJECT_FACT
```

例子：

```text
记住：这个项目测试命令是 mvn test。
```

## 状态策略

当前状态使用已有枚举：

```text
PENDING
ACTIVE
REJECTED
DISABLED
CONFLICT
```

这里的 `PENDING` 就是设计文档里说的候选记忆。

第一版策略：

```text
明确“记住” + 低风险 + 无冲突
  -> ACTIVE

普通“以后/后续/每次”类表达
  -> PENDING

同 normalizedKey 已有 ACTIVE，且内容不同
  -> CONFLICT

疑似包含密钥、token、密码
  -> 跳过，不保存
```

## 冲突策略

第一版冲突很保守。

判断方式：

```text
同一个 workspace + user + normalizedKey 下
如果已经存在 ACTIVE 记忆
新记忆内容不同
则新记忆进入 CONFLICT
并写 memory_conflicts
```

例子：

```text
旧 ACTIVE：
以后先讲设计再写代码。

新候选：
以后不要解释，直接改代码。

结果：
新记忆进入 CONFLICT，等待人工处理。
```

## 运行事件

记忆捕获完成后，会把统计结果放到 `RUN_FINISHED` metadata：

```text
memoryCaptured
memoryActivated
memoryPending
memoryConflicts
```

这样测试时可以看：

```text
agent_events 里的 RUN_FINISHED
```

也可以直接查：

```text
memory_entries
memory_conflicts
```

## 当前测试方式

### 自动生效

输入：

```text
记住：以后改代码必须写中文注释。
```

预期：

```text
memory_entries 新增一条
memory_type = WORKING_STYLE
status = ACTIVE
```

下一次 Agent 调用时，它会进入 system prompt 的长期记忆区。

### 候选记忆

输入：

```text
以后这个项目不要修改 generated 目录。
```

预期：

```text
memory_entries 新增一条
memory_type = PROJECT_CONSTRAINT
status = PENDING
```

因为用户没有明确说“记住”，第一版先等审核。

### 冲突记忆

先输入：

```text
记住：以后先讲设计再改代码。
```

再输入：

```text
记住：以后不要讲设计，直接改代码。
```

预期：

```text
第二条进入 CONFLICT
memory_conflicts 新增记录
```

## 面试表达

可以这样说：

```text
长期记忆不能直接等于 prompt。
我们把长期记忆做成独立生命周期：捕获、候选、审核、冲突、ACTIVE 注入。

运行前只注入 ACTIVE 记忆，并做数量和字符预算控制，避免长期记忆无限增长污染上下文。
运行后先用规则捕获明确的用户偏好、项目约束和项目事实。

用户明确说“记住”的低风险记忆可以自动 ACTIVE；
普通“以后/后续”类表达进入 PENDING；
同 key 已有 ACTIVE 且内容不同的新记忆进入 CONFLICT。

这样既能让 Agent 逐步学习用户和项目，又不会让未审核或冲突信息直接影响模型行为。
```
