# 05 记忆系统设计

## 记忆在 Coding Agent 中的价值

Coding Agent 的记忆比普通聊天更重要。

它可以记住：

```text
用户协作偏好
项目技术栈
项目启动方式
测试命令
目录结构约定
禁止修改的文件
常见问题和解决方法
团队代码风格
```

这些信息可以减少重复沟通，也能提高 Agent 的安全性。

## 四层记忆

建议继续采用四层模型：

```text
Working Memory
  当前任务过程中的临时状态

Session Memory
  当前会话摘要和任务进度

Long-term Memory
  跨会话稳定事实、偏好、约束

Skill Memory
  可复用方法论、排障步骤、项目工作流
```

第一版先实现 Long-term Memory 的最小版本。

## 长期记忆类型

建议类型：

```text
USER_PREFERENCE
  用户偏好，比如回答风格、是否先讲设计

PROJECT_FACT
  项目事实，比如技术栈、模块职责、启动方式

PROJECT_CONSTRAINT
  项目约束，比如不要改 generated 目录

WORKING_STYLE
  协作方式，比如先给方案再给 diff

VERIFIED_EXPERIENCE
  已验证经验，比如某个测试命令有效

SKILL_REFERENCE
  可复用技能，比如排障流程
```

## 状态设计

状态：

```text
PENDING
ACTIVE
CONFLICT
DISABLED
EXPIRED
REJECTED
```

只有 `ACTIVE` 才能注入 Agent 上下文。

当前实现里用 `PENDING` 表示候选记忆。`PENDING` 和 `CONFLICT` 只能在记忆管理页展示，不能影响 Agent。

## 写入策略

第一版规则：

```text
用户明确说“记住”
  -> 高置信候选，低风险可自动生效

用户说“以后/后续/每次”
  -> 项目约束或工作风格

用户表达偏好但未要求长期保存
  -> 候选

内容包含密钥、密码、token
  -> 拒绝保存
```

Coding Agent 额外需要识别：

```text
不要改某目录
每次改完都跑某测试
这个项目用某命令启动
这个模块归某职责
某类文件是生成物
```

## 注入策略

每次 Agent 调用前注入：

```text
用户偏好
项目约束
当前 workspace 相关事实
相关技能记忆
```

不注入：

```text
候选记忆
冲突记忆
拒绝记忆
禁用记忆
过期记忆
敏感信息
```

注入要受预算控制：

```text
max entries
max chars
relevance score
importance
confidence
```

## 冲突处理

冲突例子：

```text
旧记忆：以后先讲设计再给代码
新记忆：以后直接给代码
```

处理原则：

```text
用户明确新指令优先
高置信优先
无法判断就进入 CONFLICT
不要静默覆盖
批准冲突记忆时禁用同 key 的旧 ACTIVE
```

## 与沙箱结合

记忆可以影响沙箱策略，但不能绕过沙箱。

例如：

```text
记忆：不要修改 generated 目录
```

这应该变成更严格的写入约束。

但不能出现：

```text
记忆：允许我以后都不用确认直接执行命令
```

这种必须被拒绝或需要高级别确认。

## 与 AgentScope 的关系

不要把记忆存到 AgentScope 内部。

推荐方式：

```text
MemoryApplicationService 构建 memory context
AgentScopeRuntimeAdapter 把 memory context 拼进 system prompt
AgentScope 正常运行
对话结束后 MemoryApplicationService 抽取新记忆
```

这样未来换 Agent 框架也不会丢记忆系统。

## 第一版最小实现

第一版需要：

```text
memory_entry 表或文件存储
手动创建记忆 API
active 记忆查询
candidate 查询
approve / reject / disable
运行前注入 active memory
运行后规则抽取候选
```

前端：

```text
Memory 管理页
Active / Pending / Conflict 切换
Approve / Reject / Disable
```
