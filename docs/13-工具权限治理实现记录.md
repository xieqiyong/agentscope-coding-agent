# 工具权限治理实现记录

## 1. 这次解决的问题

工具权限治理解决的是：大模型想调用工具时，平台如何判断“能不能直接执行、是否需要用户确认、是否要拒绝”。

它不是沙箱的替代品。沙箱负责最终安全边界，例如路径不能逃出工作区、敏感文件不能读写；权限治理负责运行时决策，例如直接写文件前是否暂停，让用户确认后再继续。

## 2. 当前策略

当前策略分为三类：

- 读取类工具：直接放行，例如 `Read`、`Grep`、`Glob`、`LS`、`WebSearch`。
- 提案类工具：直接放行，例如 `propose_patch`、`propose_file_change`，因为它们只保存提案，不直接写入磁盘。
- 直接写入类工具：默认需要确认，例如 `write_file`、`Write`、`Edit`、`apply_patch`。

直接写入确认通过配置控制：

```yaml
agent:
  runtime:
    permission:
      enabled: true
      direct-write-approval-enabled: true
      approval-required-tools:
        - write_file
        - Write
        - Edit
        - apply_patch
```

如果学习或调试阶段想恢复“同工作区直接写入”，可以把 `direct-write-approval-enabled` 关掉。即使关闭，sandbox 仍然会做路径和敏感文件校验。

## 3. AgentScope 对应能力

这次接入的是 AgentScope 自带的权限机制：

- `PermissionContextState`：描述当前 Agent 的权限上下文。
- `PermissionRule`：描述某个工具的 allow、ask、deny 规则。
- `PermissionEngine`：工具执行前进行权限判断。
- `RequireUserConfirmEvent`：需要用户确认时发出的事件。
- `ConfirmResult`：用户确认或拒绝后，恢复 AgentScope 执行所需的结果。
- `enablePendingToolRecovery(true)`：允许 AgentScope 从等待工具确认的位置继续执行。

当前平台把 AgentScope 的 `ASK` 映射成自己的审批流程。

## 4. 完整生命周期

1. 用户发起聊天。
2. 平台创建 `conversation_messages` 用户消息。
3. 平台创建 `agent_runs`，状态为 `RUNNING`。
4. 平台构建 AgentScope `ReActAgent`，注入工具、sessionKey、permissionContext。
5. 模型生成工具调用。
6. AgentScope 在执行工具前检查 `PermissionEngine`。
7. 如果工具需要确认，AgentScope 发出 `RequireUserConfirmEvent`。
8. 平台把事件翻译成 `CONFIRMATION_REQUIRED`。
9. 平台为每个待确认工具调用写入一条 `approval_requests`。
10. `agent_runs.status` 从 `RUNNING` 变为 `WAITING_APPROVAL`。
11. SSE 把 `approvalId`、工具名、风险等级和 runId 推给前端。
12. 前端显示工具确认卡片。
13. 用户点击批准或拒绝。
14. 前端调用 `approval-stream`。
15. 后端更新 `approval_requests.status` 为 `APPROVED` 或 `REJECTED`。
16. 后端用审批详情重建 AgentScope `ToolUseBlock`。
17. 后端构造 `ConfirmResult`，放入消息 metadata。
18. AgentScope 从原 sessionKey 恢复等待中的工具调用。
19. 如果批准，工具继续执行；如果拒绝，AgentScope 获得“用户拒绝”的工具结果。
20. Agent 继续推理，直到完成、失败、超时或再次等待确认。

## 5. 状态流转

工具权限治理会使用已有运行状态机：

```text
RUNNING
  ↓ 需要用户确认
WAITING_APPROVAL
  ↓ 用户批准或拒绝
RUNNING
  ↓ 正常完成
COMPLETED
```

异常时：

```text
RUNNING / WAITING_APPROVAL
  ↓ 异常
FAILED / TIMEOUT / CANCELLED
```

终态优先。也就是说，一旦 run 已经进入 `COMPLETED`、`FAILED`、`TIMEOUT`、`CANCELLED`，后续迟到的恢复请求不能再把它改回 `RUNNING`。

## 6. 表设计职责

`agent_runs`：记录一次 Agent loop 的整体状态。

`agent_events`：记录运行事件流，用于前端回放、排障和可观测性。

`approval_requests`：记录一次用户确认请求，包括工具调用详情、审批状态、审批人和审批时间。

`tool_calls`：用于记录工具实际调用轨迹。当前主要由事件流重建，后续可以把工具调用开始、参数、结果更完整地落到这张表。

## 7. 和 LangGraph 的关系

思想上基本对应：

- AgentScope `RequireUserConfirmEvent` 对应 LangGraph `interrupt`。
- AgentScope `SessionKey + enablePendingToolRecovery` 对应 LangGraph `thread_id + checkpoint`。
- AgentScope `ConfirmResult` 对应 LangGraph `Command(resume=...)`。
- 平台的 `approval_requests` 对应产品层的人审任务和审计记录。

区别是：LangGraph 更强调图节点恢复，AgentScope 当前更像 ReActAgent 内部工具调用恢复。无论用哪个框架，真正的平台级治理都还要自己设计审批表、前端确认、用户权限、审计和超时处理。

## 8. 当前边界

当前版本已经跑通工具确认主链路，但还有几个后续可以继续完善的点：

- 工具风险等级目前按工具名静态判断，后续可以结合参数判断，例如路径、文件类型、改动行数。
- 多个工具同时等待确认时，目前每个工具一条审批记录，前端先按单条审批恢复。
- 审批超时、撤销、批量批准还没有产品化。
- `tool_calls` 表还可以进一步记录每次工具调用的完整参数、结果和审批关联。
- 老的 AgentScope session 可能没有新的 permissionContext，测试权限治理时建议新建会话或清理对应 sessionKey。
