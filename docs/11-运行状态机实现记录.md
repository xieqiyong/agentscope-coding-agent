# 11 运行状态机实现记录

## 这篇文档解决什么

这篇文档记录我们在 checkpoint 之后补的第一版运行状态机。

核心目标是把这句话落到代码和数据库里：

```text
checkpoint 保存恢复现场，状态机决定当前允不允许恢复、暂停、完成、失败、超时或取消。
```

## 为什么 checkpoint 不能替代状态机

checkpoint 解决的是：

```text
AgentScope 内部执行到哪里了
当前上下文是什么
有没有 pending tool
下一步从哪里继续
```

状态机解决的是：

```text
这次 Agent Run 当前是什么业务状态
是否允许继续执行
是否允许等待用户确认
是否已经终态
是否允许旧 checkpoint 再次恢复
前端应该展示什么运行状态
数据库审计应该如何解释这次运行
```

所以二者关系是：

```text
状态机 = 生命周期规则
checkpoint = 恢复现场数据
```

恢复时正确顺序是：

```text
1. 先查 agent_runs.status
2. 判断当前状态是否允许恢复
3. 允许恢复时，再加载 Redis AgentScope checkpoint
4. 继续 Agent Loop
```

## 第一版状态集合

当前第一版使用这些状态：

```text
RUNNING
  Agent Run 正在执行。

WAITING_APPROVAL
  Agent Run 暂停，等待用户确认。

COMPLETED
  Agent Run 正常完成。

FAILED
  Agent Run 异常失败。

TIMEOUT
  Agent Run 超过 timeoutSeconds。

CANCELLED
  Agent Run 被用户或系统取消。
```

终态是：

```text
COMPLETED
FAILED
TIMEOUT
CANCELLED
```

终态的关键规则：

```text
一旦进入终态，后续晚到的完成、失败、确认事件不能再覆盖它。
```

这样可以避免：

```text
用户已经取消了，后台任务又把 run 改成 COMPLETED。
run 已经 TIMEOUT，后面模型迟到结果又把它改成 COMPLETED。
旧 checkpoint 还在 Redis，系统误以为可以继续执行。
```

## 合法流转

当前允许：

```text
RUNNING -> WAITING_APPROVAL
RUNNING -> COMPLETED
RUNNING -> FAILED
RUNNING -> TIMEOUT
RUNNING -> CANCELLED

WAITING_APPROVAL -> RUNNING
WAITING_APPROVAL -> COMPLETED
WAITING_APPROVAL -> FAILED
WAITING_APPROVAL -> TIMEOUT
WAITING_APPROVAL -> CANCELLED
```

为什么允许 `WAITING_APPROVAL -> COMPLETED`：

```text
第一版还没有完整接入 approval/resume。
有些框架事件可能先出现确认事件，但实际流还会继续结束。
为了不让第一版状态机把主链路卡死，暂时允许它直接收口。
后续接完整 pending action 后，可以把这个流转收紧。
```

## 代码分层

第一版分成两层。

### AgentRunStateMachine

职责：

```text
只判断 A 状态能不能进入 B 状态
不操作数据库
不处理事件
不调用 AgentScope
```

它表达的是状态机规则。

### AgentRunLifecycleService

职责：

```text
创建 agent_runs
修改 agent_runs.status
写 finished_at
写 error_message
写 input_tokens / output_tokens
保证终态优先
```

也就是说：

```text
以后 agent_runs.status 不应该散落在业务代码里直接 set。
所有 run 状态修改都应该走生命周期服务。
```

## 主链路如何接入

一次请求开始：

```text
保存用户消息
  -> AgentRunLifecycleService.startRun(...)
  -> agent_runs.status = RUNNING
  -> 发 RUN_STARTED
```

AgentScope 正常完成：

```text
保存助手消息
  -> AgentRunLifecycleService.completeRun(...)
  -> agent_runs.status = COMPLETED
  -> 写 input_tokens / output_tokens / finished_at
  -> 发 RUN_FINISHED
```

AgentScope 抛异常：

```text
如果异常链路里包含 timeout
  -> AgentRunLifecycleService.timeoutRun(...)
  -> agent_runs.status = TIMEOUT

否则
  -> AgentRunLifecycleService.failRun(...)
  -> agent_runs.status = FAILED
```

AgentScope 发确认事件：

```text
CONFIRMATION_REQUIRED
  -> AgentRunLifecycleService.waitForApproval(...)
  -> agent_runs.status = WAITING_APPROVAL
  -> 发 RUN_STATUS_CHANGED

CONFIRMATION_RESULT
  -> AgentRunLifecycleService.resumeFromApproval(...)
  -> agent_runs.status = RUNNING
  -> 发 RUN_STATUS_CHANGED
```

## RUN_STATUS_CHANGED 事件

这次新增了平台事件：

```text
RUN_STATUS_CHANGED
```

用途：

```text
记录状态机层面的状态变化
推给前端右侧运行事件面板
写入 agent_events，方便后续回放和排障
```

它不作为聊天正文。

前端聊天气泡仍然只关心：

```text
ANSWER_DELTA
THINKING_*
TOOL_*
RUN_FINISHED
RUN_ERROR
```

## 和 LangGraph 的对应关系

LangGraph 把状态、节点、边和 checkpoint 做成图执行框架。

对应关系可以这样理解：

```text
LangGraph State
  对应我们的 RuntimeContext + AgentScope AgentState + 部分数据库状态

LangGraph Node
  对应模型调用、工具调用、审批节点、最终回答节点

LangGraph Edge
  对应状态机流转规则

LangGraph Checkpointer
  对应我们的 Redis AgentScope Session

LangGraph interrupt
  对应我们的 WAITING_APPROVAL + approval_requests + checkpoint resume
```

区别是：

```text
LangGraph 是图驱动状态机。
我们当前是 Agent Loop 外围补 Runtime 状态机。
```

## 当前边界

第一版已经完成：

```text
agent_runs.status 统一入口
RUNNING / WAITING_APPROVAL / COMPLETED / FAILED / TIMEOUT / CANCELLED
终态优先
确认事件驱动 WAITING_APPROVAL / RUNNING
超时异常识别为 TIMEOUT
RUN_STATUS_CHANGED 事件落库和前端展示
```

第一版还没有完成：

```text
真正的 approval resume API
用户点击确认后恢复 pending tool
运行中取消并中断后台线程
多实例并发恢复锁
approval_requests 和 WAITING_APPROVAL 的强绑定
```

这些后续应放到：

```text
权限治理
Checkpoint + Pending Action
运行环境治理
```

## 面试表达

可以这样说：

```text
我们把 Agent Run 的生命周期从执行代码里抽出来，做成一个独立状态机。
状态机只负责判断 RUNNING、WAITING_APPROVAL、COMPLETED、FAILED、TIMEOUT、CANCELLED 之间是否合法流转。

真正修改数据库状态由 AgentRunLifecycleService 统一完成。
这样可以避免业务代码到处 setStatus，也能保证终态优先，防止后台任务晚到结果覆盖用户取消或超时。

Redis checkpoint 仍然只负责保存 AgentScope 内部执行现场。
系统恢复时先查 MySQL 的 agent_runs.status，判断是否允许恢复；
只有状态允许时，才用 sessionKey 加载 Redis checkpoint 继续 Agent Loop。
```
