# 09 Agent 运行体验优化记录

## 本次优化的问题

本次针对三个体验问题：

```text
1. 前端没有展示思考过程。
2. Agent 有时读了很少代码就下结论。
3. Agent 好像不管问什么都要先调用工具。
```

这三个问题本质上都和 Agent Runtime 的可观测性和工具策略有关。

## 思考过程展示

AgentScope 会产生 thinking 相关事件：

```text
THINKING_STARTED
THINKING_DELTA
THINKING_FINISHED
```

前端需要把它们转成消息里的 thinking 状态，而不是丢弃。

当前采用类似 ChatGPT 的展示方式：

```text
正在思考
已完成思考
```

默认只展示状态和耗时，不直接展示完整推理链。

如果模型或网关没有返回原生 ThinkingBlock，前端会用 `MODEL_CALL_STARTED` / `MODEL_CALL_FINISHED` 合成思考状态，避免用户看到长时间空白。

原因：

```text
1. 用户需要知道 Agent 不是卡死，而是在分析。
2. 完整推理链可能很长，也可能包含不适合直接暴露的中间内容。
3. 工具调用和最终回答比完整推理文本更适合审计。
```

如果后续要调试完整 thinking，可以通过请求参数打开：

```text
traceThinkingContent = true
```

## 工具调用策略优化

之前系统提示词过于强调“先读取文件再回答”，容易导致 Agent 不管什么问题都先调用工具。

现在改成先判断问题类型：

```text
通用 Java 概念 / 面试题 / 学习路线 / 普通聊天
  -> 直接回答，不调用工具。

当前项目 / 当前代码 / 某个文件 / 某个接口 / 报错排查 / 代码修改
  -> 必须先调用工具获取证据。

用户明确说不用读代码、先给方案
  -> 先直接回答，除非缺少关键信息。

最新外部信息
  -> 使用 WebSearch。
```

这比“永远先调用工具”更像成熟 Coding Agent。

## 证据充分性规则

为了避免“没读完就下结论”，增加了证据规则：

```text
1. 对项目代码下结论前，要说明依据来自哪些文件或搜索结果。
2. 修改代码前，必须读取目标文件。
3. 涉及调用链、配置、测试或前后端联动时，要读取相关文件。
4. 证据不足时，不要强行下结论，要继续查或说明当前判断范围。
```

这个规则不是让 Agent 无限读文件，而是要求它在结论里标明证据边界。

正确体验应该是：

```text
基于已读取的 xxxController.java、xxxService.java，我判断...
如果还要确认完整调用链，需要继续看 xxxConfig.java 或测试文件。
```

## 迭代次数优化

前端之前固定传：

```text
maxIterations = 5
```

这会覆盖 Agent 表里的配置，导致 ReActAgent 很容易读两三个文件就达到上限。

现在去掉前端硬编码，让后端使用 Agent 配置：

```text
agents.max_iterations
```

表结构默认值是：

```text
8
```

这样更符合主流做法：运行预算应该属于 Agent 配置，不应该被普通聊天输入框写死。

## 当前效果预期

优化后应该看到：

```text
1. 流式回复开始前，会出现“正在思考”。
2. 需要读项目代码的问题，会先搜索/读取相关文件。
3. 普通概念问题不会强行调用工具。
4. 项目代码结论会更倾向于说明依据。
5. 复杂任务比之前更不容易过早触发最大循环次数。
```

## 后续还能继续做

后面可以继续增强：

```text
1. 给工具调用增加“证据清单”聚合。
2. 给回答加自动 evidence footer。
3. 用 MiddlewareBase 做工具预算和上下文预算治理。
4. 引入任务计划状态，让 Agent 先声明检查计划再执行。
5. 对不同任务类型使用不同 maxIterations。
```
