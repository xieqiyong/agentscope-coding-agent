# 命令级沙箱实现记录

这份文档记录 Bash/Shell 工具第一轮实现。目标不是让大模型随便执行系统命令，而是把“命令执行”纳入 Agent Runtime 的权限治理、工作区沙箱和事件审计。

## 为什么不能直接开放 Shell

Shell 是 Coding Agent 里风险最高的一类工具，因为它可以绕过文件工具的细粒度限制：

- 可以删除文件。
- 可以启动交互进程导致运行卡死。
- 可以读取系统环境变量和敏感文件。
- 可以通过管道、重定向、下载脚本等方式扩大影响范围。
- 可以脱离 workspace root 操作机器上的其他目录。

所以第一原则是：

```text
大模型可以提出命令
平台先请求用户确认
用户批准后仍然要过命令沙箱
命令沙箱通过后才真正执行
```

## 当前执行链路

```text
用户发起任务
  ↓
AgentScope ReActAgent 进入循环
  ↓
模型决定调用 Bash / Shell / run_command
  ↓
RuntimeToolGuard 命中 ASK 规则
  ↓
RuntimeEvent: CONFIRMATION_REQUIRED
  ↓
approval_requests 落库
  ↓
前端展示确认卡片
  ↓
用户批准
  ↓
approval-stream 读取平台 pending action
  ↓
平台直接执行挂起命令
  ↓
CommandSandboxService 再次校验命令
  ↓
ProcessBuilder 执行
  ↓
返回 stdout / stderr / exitCode / timeout / truncated
  ↓
平台生成最终答复并结束 run
```

关键点：

```text
ToolGuard 权限确认解决“用户是否允许这次工具调用”
命令沙箱解决“这条命令是否真的安全可执行”
两者不能互相替代
```

## 一次关键修正：ASK 不能当普通工具结果

平台 ToolGuard 第一版曾经在需要审批时返回普通文本：

```text
平台已暂停工具执行，等待用户确认
```

这个做法会带来一个问题：

```text
工具结果被继续喂给模型
模型又生成“我还在等你批准”
用户批准后，旧回答还停留在页面上
```

正确做法是把审批当作真正的 runtime interrupt：

```text
RuntimeToolGuard 发出 CONFIRMATION_REQUIRED
工具抛出 ToolSuspendException
AgentScope 本轮停止继续推理
run 进入 WAITING_APPROVAL
approval-stream 执行被批准的工具
工具完成后平台发出最终 ANSWER_DELTA 收口
```

这更接近 LangGraph 的 `interrupt` / `resume` 思想：

```text
interrupt 不是让模型解释“我被暂停了”
interrupt 是运行时暂停现场，等待外部决策后恢复
```

同时，审批通过不代表工具一定成功。审批只是第一道门，命令沙箱是第二道门：

```text
用户批准 Bash
  ↓
命令沙箱校验
  ↓
允许则执行
拒绝则返回 ERROR 工具结果，并生成明确最终答复
```

## 第一轮开放的工具

当前注册了 3 个工具名：

- `Bash`
- `Shell`
- `run_command`

它们本质是同一个能力，只是为了兼容不同 Coding Agent 的工具命名习惯。

工具参数：

- `command`：单行命令。
- `description`：执行目的说明，方便审计和前端展示。
- `timeoutSeconds`：超时时间，会被沙箱最大值限制。
- `workingDirectory`：workspace 相对目录，默认 workspace 根目录。

## 沙箱策略

### 1. 工作目录限制

命令只能在当前 workspace 内执行。

如果模型传入绝对路径或相对路径，平台都会先解析成真实路径，并判断是否仍然位于 workspace root 内。

禁止：

```text
../
符号链接逃逸
不存在的工作目录
非目录工作目录
```

### 2. 允许列表

第一轮采用 allowlist 前缀策略，只开放常见测试、构建和只读 git 命令。

默认允许示例：

```text
mvn test
mvn -q test
mvn -q -DskipTests compile
mvn -q -DskipTests package
npm test
npm run test
npm run build
pnpm test
pnpm run build
yarn test
yarn build
git status
git diff
git diff --stat
```

这样做的好处是简单、可解释、面试里也容易讲清楚：

```text
先做最小可用命令集合
后续根据审计数据逐步扩展 allowlist
而不是一开始就做全功能 Shell
```

### 3. 禁止危险 shell 形态

第一轮直接拒绝这些 shell 控制符：

```text
&&
||
;
|
`
$(
>
<
```

原因是这些控制符会让“单条命令”变成“命令组合”：

- 管道可能变成 `curl | sh`。
- 重定向可能覆盖文件。
- `&&` / `;` 可能串联危险命令。
- `$()` 和反引号可能做命令替换。

### 4. 禁止危险片段

默认拒绝：

```text
rm
rmdir
del
format
shutdown
reboot
git reset --hard
git clean
curl |
wget |
Invoke-WebRequest
Invoke-Expression
powershell -EncodedCommand
Start-Process
```

### 5. 禁止交互式命令

默认拒绝：

```text
vim
vi
nano
less
more
top
htop
ssh
mysql
psql
redis-cli
node
python
powershell
cmd
```

原因是交互式命令会等待用户输入，容易让 Agent Run 长时间挂死。

### 6. 超时和输出截断

每条命令都有超时限制。

输出分为：

- stdout
- stderr

两者分别截断，避免大输出撑爆事件流、数据库或前端页面。

工具结果会返回：

```text
命令
目的
工作目录
执行状态
退出码
耗时
stdout 是否截断
stderr 是否截断
STDOUT
STDERR
```

## 和权限治理的关系

命令工具被标记为高风险工具：

```text
Bash        CRITICAL
Shell       CRITICAL
run_command CRITICAL
```

当前策略：

```text
读取类工具默认放行
写文件类工具 HIGH，需要确认
命令类工具 CRITICAL，由平台 RuntimeToolGuard 确认
```

用户批准后，不代表一定执行成功。

还有第二道门：

```text
CommandSandboxService
```

如果命令不在 allowlist，仍然返回：

```text
执行状态：REJECTED
拒绝原因：命令不在允许列表内
```

## 和 AgentScope 的关系

AgentScope 负责：

- 运行 ReActAgent loop。
- 让模型决定是否调用工具。
- 产生工具调用、确认、工具结果等事件。
- 对写文件类工具仍可保留 PermissionEngine 兜底。

平台负责：

- 注册受控工具。
- 对命令类工具执行 RuntimeToolGuard。
- 发出平台级确认事件。
- 把确认事件落库为 pending action。
- 让前端展示确认卡片。
- 用户决策后通过 approval-stream 直接执行挂起命令。
- 命令执行前做沙箱校验。
- 持久化 Agent Run 和 Runtime Event。

核心边界：

```text
AgentScope 是 Agent Runtime
平台是治理层和产品层
```

## 当前没有做的事

第一轮还没有做：

- stdout/stderr 实时流式返回。
- 后台任务管理。
- 进程树清理。
- 环境变量白名单。
- 网络访问策略。
- 命令级审计详情表。
- 基于项目类型的动态 allowlist。
- Windows / Linux / macOS 差异化命令模板。

这些属于后续“治理增强”阶段。

## 面试讲法

可以这样概括：

```text
我没有直接把 Shell 暴露给大模型，而是做成三层治理：

第一层：平台 RuntimeToolGuard
模型调用 Bash 进入工具方法后，平台先触发 ASK，前端让用户确认。

第二层：平台审批和状态机
确认请求落库，Agent Run 进入 WAITING_APPROVAL，用户批准后通过 approval-stream 读取 pending action。

第三层：命令沙箱
即使用户批准，也必须通过 workspace root、allowlist、危险片段、交互命令、超时和输出截断校验。

这样既能保留 Coding Agent 的自动化能力，又能把风险控制在可审计、可恢复、可扩展的范围内。
```

## 后续优化方向

下一轮治理增强可以继续做：

- 命令执行实时事件流。
- 命令进程树 kill。
- 命令审计表。
- allowlist 按 workspace 技术栈动态生成。
- 用户长期偏好影响命令策略。
- 高风险命令二次确认。
- 文件变更前后快照。
- 失败补偿和回滚。
